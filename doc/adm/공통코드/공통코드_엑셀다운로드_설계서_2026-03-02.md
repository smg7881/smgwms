## 2. 기존 구현 화면 분석 (사용자관리·부서관리)

### 2.1 호출 흐름 (정상 동작 근거)

사용자관리(`system/users`)와 부서관리(`system/dept`)는 동일한 패턴으로 구현되어 있다.

1. `app/components/system/users/page_component.html.erb`
   - `excel_action: "click->user-crud#downloadExcel"`를 `Ui::GridHeaderComponent`에 전달
   - `Ui::GridHeaderComponent`는 내부적으로 항상 `Ui::GridActionsComponent`를 렌더링
   - `excel_action`이 지정되면 그것을 사용하고, 미지정 시 CSV fallback(`grid-actions#exportCsv`)으로 자동 전환
2. `app/javascript/controllers/user_crud_controller.js`
   - `BaseCrudController` 상속, `excelExportUrl` value 선언
3. `app/javascript/controllers/base_crud_controller.js`
   - `downloadExcel()` 공통 메서드에서 `window.location.href = this.excelExportUrlValue`
4. `app/controllers/system/users_controller.rb`
   - `include System::ExcelTransferable`
   - `excel_resource_key`, `excel_export_scope` 구현
5. `app/controllers/concerns/system/excel_transferable.rb`
   - `excel_export` 공통 로직으로 `.xlsx` 생성/응답
   - `excel_template` (템플릿 다운로드), `excel_import` (업로드 처리)도 동일 concern에 포함
6. `app/services/excel/handler_registry.rb` + `app/services/excel/handlers/users_handler.rb`
   - `"users"` 키로 핸들러 등록, 헤더/행 데이터 규격화

부서관리는 위와 동일하며 `"dept"` 키로 `DeptHandler`가 등록되어 있다.
(`routes.rb`에 `excel_template`, `excel_export`, `excel_import` 라우트 모두 포함)

### 2.2 컨트롤러 상속 구조

프로젝트에는 두 가지 독립적인 Stimulus 베이스 컨트롤러 계열이 존재한다.

```
Stimulus Controller
  ├── BaseCrudController    (모달 CRUD 화면)
  │     ├── user_crud_controller
  │     ├── dept_crud_controller
  │     └── (기타 모달 CRUD 화면들)
  │
  └── BaseGridController    (인라인 그리드 편집 화면)
        ├── code_grid_controller
        └── (기타 그리드 화면들)
```

- `BaseCrudController`: Excel 5개 메서드(`downloadExcel`, `downloadExcelTemplate`, `openImportHistory`, `openExcelImport`, `submitExcelImport`) 직접 구현
- `BaseGridController`: Excel 메서드 없음

### 2.3 공통화 수준 판정 (사용자관리·부서관리 기준)
- 공통화 완료 영역:
  - 서버 엑셀 생성 로직 (`ExcelTransferable`, `WorkbookBuilder`, `Handler`)
  - 프론트 기본 다운로드 메서드 (`BaseCrudController#downloadExcel`)
- 부분 공통화 영역:
  - 화면별로 `excel_action`을 수동 지정해야 함
  - 리소스별로 `HandlerRegistry`에 수동 등록 필요
  - `BaseGridController` 계열에 Excel 메서드 없음 (양 계열 간 중복/불일치)

결론: 사용자관리·부서관리는 실사용 관점에서 정상이나, 두 베이스 컨트롤러 간 Excel 공통화가 분리되어 있어 표준화가 미흡하다.

## 3. 공통코드 화면 현황

- `system/code`는 현재 그리드 우측 상단 엑셀 아이콘이 CSV fallback(`grid-actions#exportCsv`)으로 동작
  - `Ui::GridHeaderComponent`가 `excel_action` 미지정 시 `Ui::GridActionsComponent` 내부에서 자동으로 CSV fallback 처리
- 원인:
  - `code` 화면에서 `excel_action` 미지정
  - `code-grid`는 `BaseCrudController`가 아니라 `BaseGridController` 계열이라 `downloadExcel` 훅이 없음
  - `System::CodeController`/`CodeDetailsController`에 excel export 엔드포인트 없음
  - `HandlerRegistry`에 code 관련 handler 없음

결론: `system/code` 엑셀다운로드는 별도 개발 필요.

## 4. 공통화 미흡 항목 정리

### 4.1 프론트엔드
- `Ui::GridActionsComponent`가 fallback(CSV)은 공통이나, Excel 액션 연결은 화면별 수동 주입
- `BaseGridController` 계열에 Excel 다운로드 공통 메서드 부재
- `BaseCrudController`와 `BaseGridController` 양쪽에 Excel 로직이 분리되어 있어 향후 공통 수정 시 두 곳을 동시에 관리해야 함

### 4.2 백엔드
- `HandlerRegistry`가 `case` 기반 수동 확장 구조
- 리소스 추가 시 Controller + Route + Handler 등록을 각각 수동 반영해야 함

## 5. 공통화 변경 설계

### 5.1 설계 목표
- 사용자관리·부서관리에서 검증된 Excel 다운로드 흐름을 `system/code`에도 동일하게 적용
- `BaseCrudController`와 `BaseGridController` 양쪽이 동일한 Excel mixin을 사용하여 단일 표준으로 통일
- 향후 신규 화면도 최소 변경으로 Excel 다운로드를 붙일 수 있게 공통화 수준 상향

### 5.2 1차 적용안 — 생략
1차 적용안(현재 구조 유지 빠른 적용)은 진행하지 않는다. 표준 수립 목적에 따라 곧바로 2차 공통화안을 적용한다.

### 5.3 2차 공통화안 (채택, 전체 구현)

#### 5.3.1 프론트엔드 — ExcelDownloadable mixin 분리

프로젝트 기존 mixin 패턴(`Object.assign(Class.prototype, Mixin)`)을 따라 Excel 메서드를 독립 파일로 분리한다.

**신규 파일: `app/javascript/controllers/concerns/excel_downloadable.js`**

```javascript
export const ExcelDownloadable = {
  // 엑셀 다운로드: excelExportUrlValue 로 이동
  downloadExcel() {
    if (this.hasExcelExportUrlValue) {
      window.location.href = this.excelExportUrlValue
    }
  },

  // 엑셀 양식 템플릿 다운로드
  downloadExcelTemplate() {
    if (this.hasExcelTemplateUrlValue) {
      window.location.href = this.excelTemplateUrlValue
    }
  },

  // 업로드 이력 페이지 이동
  openImportHistory() {
    if (this.hasImportHistoryUrlValue) {
      window.location.href = this.importHistoryUrlValue
    }
  },

  // 파일 선택 다이얼로그 오픈
  openExcelImport() {
    const fileInput = this.element.querySelector("[data-excel-import-input]")
    if (fileInput) fileInput.click()
  },

  // 파일 선택 완료 시 자동 submit
  submitExcelImport(event) {
    const input = event.target
    if (input.files.length === 0) return
    const form = input.closest("form")
    form?.requestSubmit()
    input.value = ""
  }
}
```

**`BaseCrudController` 변경:**
- `downloadExcel`, `downloadExcelTemplate`, `openImportHistory`, `openExcelImport`, `submitExcelImport` 메서드 제거
- 파일 하단에 `Object.assign(BaseCrudController.prototype, ExcelDownloadable)` 추가
- 서브클래스(`user_crud_controller`, `dept_crud_controller` 등) 변경 없음
  (이미 각자 `excelExportUrl` 등 values를 선언 중)

**`BaseGridController` 변경:**
- `static values`에 Excel 관련 values 추가:
  ```javascript
  static values = {
    batchUrl: String,
    excelExportUrl: String,       // 신규
    excelTemplateUrl: String,     // 신규
    importHistoryUrl: String      // 신규
  }
  ```
- 파일 하단에 `Object.assign(BaseGridController.prototype, ExcelDownloadable)` 추가
- `BaseGridController`를 상속하는 서브클래스 중 `...BaseGridController.values` 스프레드를 사용하는 경우 자동으로 values를 상속받음

#### 5.3.2 code_grid_controller 구현

마스터는 mixin에서 상속받은 `downloadExcel()`을 그대로 사용한다.
디테일은 `selectedCodeValue`를 URL에 포함해야 하므로 `downloadDetailExcel()`을 추가 선언한다.

```javascript
static values = {
  ...BaseGridController.values,        // excelExportUrl 포함됨
  masterBatchUrl: String,
  detailBatchUrlTemplate: String,
  detailListUrlTemplate: String,
  selectedCode: String,
  detailExcelExportUrlTemplate: String // 신규: /system/code/:code/details/excel_export
}
```

추가 메서드:
```javascript
// 마스터 엑셀: mixin의 downloadExcel() 그대로 사용 (excelExportUrlValue 사용)

// 디테일 엑셀: selectedCodeValue로 URL 치환 필요
downloadDetailExcel() {
  if (!this.selectedCodeValue) {
    showAlert("코드를 먼저 선택해주세요.")
    return
  }
  const url = buildTemplateUrl(
    this.detailExcelExportUrlTemplateValue,
    ":code",
    this.selectedCodeValue
  )
  window.location.href = url
}
```

`page_component.html.erb` 그리드 헤더 변경:
```erb
<%# 마스터 — mixin 상속 downloadExcel 사용 %>
<%= render Ui::GridHeaderComponent.new(
  ...,
  excel_action: "click->code-grid#downloadExcel"
) %>

<%# 디테일 — 커스텀 downloadDetailExcel 사용 %>
<%= render Ui::GridHeaderComponent.new(
  ...,
  excel_action: "click->code-grid#downloadDetailExcel"
) %>
```

`page_component.rb` 추가:
```ruby
def master_excel_export_url
  system_code_excel_export_path
end

def detail_excel_export_url_template
  "/system/code/:code/details/excel_export"
end
```

`page_component.html.erb` data 값 주입:
```erb
data-code-grid-excel-export-url-value="<%= master_excel_export_url %>"
data-code-grid-detail-excel-export-url-template-value="<%= detail_excel_export_url_template %>"
```

#### 5.3.3 백엔드 — Route / Controller / Handler

**Route (`config/routes.rb`):**
```ruby
resources :code, controller: :code, only: [...], param: :id do
  collection do
    get :excel_export       # 신규
  end
  post :batch_save, on: :collection
  resources :details, controller: :code_details, only: [...], param: :detail_code do
    collection do
      get :excel_export     # 신규
    end
    post :batch_save, on: :collection
  end
end
```

**`System::CodeController`:**
```ruby
include System::ExcelTransferable

private
  def excel_resource_key = :code_header

  def excel_export_scope
    headers_scope   # 기존 검색 조건 메서드 재사용
  end
```

**`System::CodeDetailsController`:**
```ruby
include System::ExcelTransferable

private
  def excel_resource_key = :code_detail

  def excel_export_scope
    # params[:code_id] 필터 필수 — 미적용 시 전체 덤프 버그 발생
    AdmCodeHeader.find_by!(code: params[:code_id]).details.ordered
  end
```

**`HandlerRegistry` 추가:**
```ruby
when "code_header" then Excel::Handlers::CodeHeaderHandler.new
when "code_detail"  then Excel::Handlers::CodeDetailHandler.new
```

#### 5.3.4 excel_import / excel_template 적용 범위

이번 구현 범위는 **export(엑셀 다운로드)만** 포함한다.
import(업로드)·template(양식 다운로드) 기능은 별도 의사결정 후 추가한다.
- 공통코드는 마스터/디테일 구조가 복잡해 import 시 부모-자식 정합성 처리가 필요
- 필요 시 `CodeHeaderHandler#import_row!`, `CodeDetailHandler#import_row!` 구현 및 라우트 추가

  - 마스터 `.xlsx` 다운로드 확인 (검색 조건 반영 여부 포함)

  - 디테일 엑셀이 선택한 코드의 상세코드만 포함하는지 확인 (전체 덤프 방지)
  - 선택 코드 없을 때 경고 처리 검증
- 공통화 검증:
  - Excel 액션 미지정 화면은 기존 CSV fallback 유지 확인

## 8. 최종 결론
- 사용자관리·부서관리 엑셀다운로드는 공통 인프라를 활용해 정상 동작한다.
- 2차 공통화 작업으로 Excel 메서드를 `ExcelDownloadable` mixin으로 분리하고, `BaseCrudController`·`BaseGridController` 양쪽에 동일하게 적용한다.
- 이 표준을 기반으로 이후 신규 화면에서도 동일한 패턴으로 Excel 기능을 추가할 수 있다.
- **디테일 `excel_export_scope` 구현 시 `params[:code_id]` 필터 적용이 필수**이며 미적용 시 전체 상세코드 덤프 버그가 발생한다.
